01좌측 nav 구조

Sidebar에 “보관함”은 워크스페이스 레벨에서 프로젝트 보관함(상단 Archive)이고, “휴지통”은 프로젝트 내부의 issue trash(?view=trash)다 — 같은 단어로 보이지만 사실 다른 레벨·다른 도메인이다. 이게 사용자에게 가장 큰 혼란이다.

crit

Archive와 Trash가 의미적으로 충돌한다

Sidebar.tsx:303 Sidebar.tsx:240 mental model

“방금 삭제한 이슈가 어디 있지? 보관함? 휴지통? 어느 프로젝트의 휴지통?”

현재 구조:

    ├── Issues
    ├── Categories

    └── Settings
제안
Workspace
├── 🏠 Home
├── 🧭 Discover
├── 📢 Announcements
└── ⭐ Project A
    ├── Issues
    ├── Categories
    ├── 📦 Archived issues     ← 같은 “지운 것” 그룹
    ├── 🗑 Trash (30d)
    └── Settings
⚙ Workspace Admin
└── Archived projects      ← 워크스페이스 단위는 admin 메뉴로

핵심: “지움”의 두 종류는 같은 위치(프로젝트 안)에 두고, “프로젝트 archive”는 워크스페이스 관리자 영역으로 옮긴다. 일반 사용자는 “프로젝트 archive”를 거의 안 쓴다 — admin이 정리할 때나 본다.

high

프로젝트 펼침 메뉴에 “Trash”가 있는데 “Archive”는 없다

Sidebar.tsx:225-243

현재 펼친 프로젝트 트리: Issues / Modules / Categories / Trash / Settings. 그런데 ProjectIssuePage에는 view=archive도 있다 (ArchiveView 컴포넌트 존재). 즉 Archive 뷰에 도달할 수 있는 사이드바 경로가 없다 — 상단 view tab으로만 갈 수 있다.

“보관”이 “휴지통”보다 자주 쓰이는 행위인데 nav에 없으니 사용자가 찾아 헤맨다.

<SubLink to={`${base}/issues?view=trash`}   icon={Trash2}

// 두 항목을 “Issues / Modules / Categories” 다음에 묶어서, 위계 시각화
med

Workspace-level Archive가 nav 최상단에 있을 이유가 없다

Sidebar.tsx:303

Home / Discover / Archived / Announcements 순서. Archived(워크스페이스 archived projects 페이지)는 거의 안 들어가는 페이지다. 상단 4개 슬롯은 “매일 1번 이상 들어가는 페이지”에 줘야 한다.

처방: Archived는 Settings → Workspace → Archived Projects로 이동. 또는 프로젝트 목록 하단 “Show archived” 토글로.

02동기화 표시 (좌측 하단)

Sidebar 하단의 WebSocket status indicator는 만들어져 있으나, 표현이 너무 나이브하다. “Connected — Real-time sync active” 풀 텍스트가 항상 보이는 건 저신뢰 신호 + 고노이즈 조합이다.

high

현재 — always visible, all states equal

med

“실시간 동기화” 라는 신뢰는 다른 곳에서 만들어야 한다

signal upgrade

연결 점은 약속이지 증거가 아니다. 사용자가 정말 안심하는 건:

  • “방금 OO이 이 카드를 옮겼습니다” 토스트 (1.5s)
  • WebSocket pulse-strip (이미 구현됨 — 더 자주 쓰여야 함)
  • Avatar 클러스터 — 같은 페이지를 보고 있는 동료 표시 (Linear, Notion)
  • Last saved 자동 저장 표시 (이슈 detail에서)

처방: 좌측 하단의 점은 “네트워크 상태”로만 격하시키고, “실시간 협업감”은 위 4가지로 분산시켜라.

03Dashboard (Workspace home)

“내 할 일”과 “최근 이슈”의 두 영역이 같은 비중으로 놓여 있다 — 그래서 어디로 시선을 둬야 할지 결정 못한다.

high

Dashboard는 “이번 주 작업”에 집중해야 한다

WorkspaceDashboard.tsxfocus

현재 정보 구조:

현재
  • 인사 + SprintProgressOrbit (큰 영역)
  • 필터 (priority / project / state group / groupBy)
  • 내 할 일 — 그룹별 (state 또는 project)
  • 최근 이슈 — 별개 섹션
제안

3-column 또는 “Today — This week — Stale” 의 3-슬롯 구조로 재구성.

  • Today: 오늘 마감 + 진행 중 + 지금 멘션 받은 것
  • This week: 이번 sprint·이번 주 마감
  • Stale: 7일+ 안 건드린 in-progress (alert)

“최근 이슈”는 dashboard 메인이 아니라 activity 사이드패널로 격하. 대시보드 1페이지에서 “지금 뭘 해야 하나”가 즉시 답이 나와야 한다.

med

필터 4종이 dashboard에 박혀 있어야 하나?

WorkspaceDashboard.tsx:200-221

priorityFilter / projectFilter / stateGroupFilter / groupBy 4개가 매번 보여진다. 대시보드는 “필터링 도구”가 아니라 “현황 요약” 페이지다. 기본은 active issues only로 자동 필터, 사용자는 “필터 변경” 버튼 한 번 누른 뒤에만 펼쳐 보이게.

low

SprintProgressOrbit은 좋은 모먼트 — 더 활용하라

brand

대시보드 우측 상단의 작은 orbit progress는 “OrbiTail다움”이 살아있는 거의 유일한 작업 페이지 모먼트다. 이걸 크게 메인 슬롯에 두고, 그 주변에 “Today / Week / Stale” 카드를 위성처럼 배치하는 것도 검토.

04Project Issue Page

한 페이지에 10개 view가 있다 — table, board, backlog, calendar, timeline, graph, sprints, analytics, archive, trash. 탭이 한 줄에 안 들어간다.

high

10개 뷰 탭을 “수평 한 줄”에 두는 건 한계

ProjectIssuePage.tsx:42-56nav scale

view 종류가 너무 많고, 의미도 다층이다 — “list/board/calendar”는 같은 데이터의 형태이고, “backlog/sprints/analytics”는 다른 뷰이고, “archive/trash”는 다른 데이터셋이다.

현재
[ Table ] [ Board ] [ Backlog ] [ Calendar ] [ Timeline ] [ Graph ] [ Sprints ] [ Analytics ] [ Archive ]
한 줄에 다 안 들어감 → wrap 또는 horizontal scroll
제안 — 3계층 분리
View: Table · Board · Calendar · Timeline · Graph
Scope: Active · Backlog · Archive
Tools: Sprints · Analytics

해석: 같은 데이터의 “보는 법(view)”과, 어떤 데이터셋을 보는가(scope), 도구(tool)를 분리. Trash는 sidebar로 이동(§01에서 제안).

med

Sprint banner와 Filter row가 따로 논다

ProjectIssuePage.tsx:300-348

스프린트 선택 시 banner가 하단에 추가되어 view tab → filter row → sprint banner → 실제 view 4단으로 쌓인다. 화면 상단 100–120px이 모두 chrome.

처방: sprint banner는 view 영역의 “in-context header”로 흡수. 또는 우측 사이드 패널로. view tab은 sticky로 두되 banner는 view 안쪽으로.

06Archive (이슈 보관함)

med

“보관함”에 “자동 정리(retention)” 정책이 없다

ArchiveView.tsxpolicy

Trash는 30일 후 자동 영구 삭제 — 좋은 정책. Archive는? 영원히 쌓인다. 6개월·1년·2년이 지난 archived 이슈가 검색·통계를 오염시킨다.

처방: Settings → Data Retention 페이지에 “Archived issues older than 1y → Trash로 이동” 옵션. 기본 비활성, 워크스페이스 admin이 켤 수 있게.

09Roadmap v3

v3는 IA 변경이라 frontend + i18n + 약간의 backend가 같이 움직인다. 작은 단위로 잘라야 함.

Sprint 1 — Quick

nav 정합성 + signal

  • 프로젝트 펼침에 Archive 추가 (§01)
  • WS status — connected는 점만, 나머지만 텍스트 (§02)
  • Trash 잔여일수 progress bar (§07)
  • Trash “Empty all” 일괄 (§07)
  • Archive 검색 input 추가 (§06)
Sprint 2–3 — Restructure

view 탭 3계층 + dashboard 재편

  • View tab을 view/scope/tool로 분리 (§04)
  • Workspace-level Archive를 admin으로 이동 (§01)
  • Dashboard “Today/Week/Stale” 슬롯 (§03)
  • BacklogView split-view: backlog → sprint drop zone (§05)
  • Settings 3계층 그루핑 (§08)
Quarter — Policy & Trust

retention + presence

  • Archive retention 정책 (1y → trash) backend + admin UI (§06)
  • archived_by 필드 + 컬럼 (§06)
  • “같은 페이지 보는 동료” avatar cluster (§02)
  • 변경 토스트 “OO이 옮겼습니다” (§02)
  • Last-saved indicator on issue detail (§02)
